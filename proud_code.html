So, you're looking for code I'm proud of eh?

I'm proud of a lot of code I've written, and I have included links to some below, but it's not just the code itself that I'm proud of.

It's the entire engineering design process behind the code.

Sure, we don't want to over-engineer things, or spend so long in the boilerplate that we never get to the functionality. But it's easy to make small mistakes when trying to code up a quick solution, and it's even easier to end up with loads of technical debt from kludges that were never meant to stick around.

As a responsible ML engineer and software developer, I always take the extra minute to consider not just the result we want now, but what we might want to have it do later on, and how easy the code is to understand and modify.

In other words, I prefer to design software systems with a holistic approach, paying mind to the scalability, realiability, security, maintainability, testability, and so on. This doesn't take a lot of extra time to do either — just some experience and the right attitude.

That being said, I know you wanted a sample of code that I've actually written myself. Although I don't have permission to upload code for most ML projects I've worked on recently, I can share a few files from my Capstone design project here:

(For more information about this project of mine, check it out at the top of my website's homepage)

  • main.py - The entry point for this program, which isn't very interesting. It mainly handles interpreting command-line arguments and opening SSH tunnels.
  • server.py - The primary load-balancing server that handles incoming user requests, dispatches them to the best available ML server, and synchronizes the databases.
  • predictor.py - The prediction engine used by each backend ML server, which trains an assortment of models on new data as it comes in (on-line learning) and makes suggestions for users when requested by the primary server.
  • selector.py - The model selector for the backend ML servers; this evaluates each model against the different types of learning tasks, and determines which model performs best (according to a given metric, e.g. accuracy) for said task.
  • models.py - A collection of a couple of simple neural networks (mostly MLPs) that are used as the default models for the backend ML servers, but these can be overridden on a per-server basis.
  • types.py - Mostly a collection of enums and dataclasses that are used throughout the codebase.
  • serial.py - Custom serialization logic for communication between the primary and backend servers.

I can't really say that I'm the proudest of this code, but I don't have too many other examples that I can freely share either. But there is also this project I worked on during my first ever machine learning job: github.com/BenChapmanKish/UG-MLRG-Project.
